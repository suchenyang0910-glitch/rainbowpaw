# OpenClaw & RainbowPaw 整合优化重构需求说明

基于对原 `RainbowPaw` 善终平台和 `rainbowpay_claw` 抽奖机器人双边 PRD 及代码的深度分析，我们发现旧版需求在**业务边界、货币系统、用户身份**上存在明显的冲突与割裂。

本说明旨在修复这些“不完善之处”，输出一份全局一致、可落地的重构需求标准。

---

## 一、 旧版需求中存在的冲突与不完善之处 (痛点分析)

1. **货币与钱包体系混乱（致命风险）**
   - **旧版缺陷**：Claw Bot 的旧 PRD 中将钱包拆分为 `cash_balance`（现金）、`credit_balance`（平台余额）和 `points_balance`（积分）。这种“美元+代币”三轨并行的设计，不仅让用户认知混乱，还会导致退款套利、提现压力大等严重的现金流风险。
   - **优化方向**：必须废弃旧钱包逻辑，全面切入**“全积分计价系统”**（`locked` + `cashable`）。

2. **双端业务边界越界（维护灾难）**
   - **旧版缺陷**：Claw Bot 的 PRD 试图在机器人内包揽“直接购买商品（Direct Buy）”的完整电商流程；而 RainbowPaw 的 PRD 也规划了完整的纪念品商城。如果两边都做购物车和订单流，会导致代码严重冗余，库存无法互通。
   - **优化方向**：严格执行“前台分离”。Claw Bot 仅保留盲盒抽奖及低客单价引流商品；复杂的“商城选购、善终履约、高价值定制”**全部强制通过 Bridge Service 深链接导流到 RainbowPawbot**。

3. **用户身份割裂（无法计算全局 LTV）**
   - **旧版缺陷**：RainbowPaw 和 Claw Bot 各自维护独立的 `users` 表，主键不同。即使用户是同一个人，平台也无法知道他在 Claw 抽了多少钱，在 RainbowPaw 消费了多少钱。
   - **优化方向**：抽离 `identity-service`，所有行为挂载在 `global_user_id` 下。

4. **拼团/分销现金流风险**
   - **旧版缺陷**：旧版拼团（3人成团返50%）及分销结算存在直接返现（cash）的倾向。
   - **优化方向**：所有返利强行拆分为 `points_locked`（防提现，逼复购）和 `points_cashable`。

---

## 二、 优化与重构后的核心业务逻辑规范

### 1. 统一的“积分制”商业闭环 (取代旧版杂乱的钱包)
- **统一定价**：废除所有 `$5`, `$10` 的前端标价。无论抽奖、商城购买、拼团返利，**一律使用“积分 (Points)”作为唯一流通货币**（后台按 2积分 = $1 美元 锚定）。
- **钱包三账户重构**：
  - `points_locked`：不可提现，仅能在 Claw 或 RainbowPaw 中消费。
  - `points_cashable`：可提现，也可消费。
  - `wallet_cash`：美元提现专用通道。
- **消费扣减强制顺序**：购买商品或抽奖时，**必须优先扣除 `points_locked`**。只有当 locked 不足时，才扣除 cashable。

### 2. 双 Bot 功能裁剪与导流 (明确边界)
- **Claw Bot（拉新/激活/变现入口）**：
  - 砍掉旧版 PRD 中的“复杂购物车”、“长尾商品 Direct Buy”逻辑。
  - 保留：`/start`、充值入口、1.5$（即3积分）抽盲盒、拼团裂变逻辑。
  - 新增：当用户触发大奖、或累计抽奖次数达标时，展示 `🌈 纪念服务/高级商城` 按钮，通过 `bridge-service` 生成带参 Token 跳转至 RainbowPawbot。
- **RainbowPawbot（高客单承接/重履约）**：
  - 砍掉：旧版中不切实际的“低门槛引流”幻想，不再做独立的裂变功能。
  - 强化：承接来自 Claw Bot 的 `global_user_id`。展示宠物档案、高端骨灰盒/定制饰品购买、以及预约线下善终服务。它直接读取中台的积分余额进行扣款。

### 3. 订单与履约状态机整合
- **抽奖极简订单 (Claw端)**：抽中实物后，仅记录 `pending_address` -> `ready_to_ship` -> `shipped`。由管理员统一导表一件代发。
- **重服务订单 (RainbowPaw端)**：继承原 RainbowPaw PRD 中的复杂状态机 (`confirmed` -> `pickup_assigned` -> `cremation_in_progress` -> `delivery`)，并联动司机与服务商系统。
- **订单数据归一**：两边生成的订单，均上报给 `report-service`，并在数据分析时合并计算为该 `global_user_id` 的总体贡献。

### 4. 动态概率与风控引擎的重构
- 旧版 Claw PRD 提到的“动态概率调整（Profit-based Adjustments）”予以保留并强化：
  - 利润率 (`profit_rate`) 的计算，**必须将用户在 RainbowPaw 的高客单消费也计算在内**。
  - 即：如果某用户在 Claw 一直抽不中，但通过导流在 RainbowPaw 花了 $200 买纪念品，该用户的 LTV 极高。风控引擎应给该用户在 Claw 端“放水”（提高超级奖概率），以此形成**双边生态的反哺**。

---

## 三、 重构实施路径调整建议 (给开发团队的 Checklist)

为了将旧代码平滑迁移到新架构，开发必须执行以下“破坏性重建”：

1. **废弃旧版 Users 表**：
   - 停止 Claw 和 RainbowPaw 独立生成 `user_id` 的逻辑。
   - 引入 `identity-service/link-user`，通过 Telegram_ID 强行合并用户。
2. **清理旧版 Wallet / Balance 逻辑**：
   - 删除 Claw 代码中的 `user_balance` 和 `cash/credit/points` 散落逻辑。
   - 全面接入 `wallet-service` 的 `earn`, `spend`, `recycle` API。
3. **前端 UI/话术大清洗**：
   - 搜索所有代码中的 `$` 符号标价，替换为 `积分` 标价。
   - 拼团文案从“返利 50%”改为“获得 X积分 (含 Locked 与 Cashable)”。
4. **引入 Bridge 服务**：
   - 在 Claw Bot 中接入 `/bridge/generate-link`。
   - 在 RainbowPaw 的 `/start` 指令中增加解析 Token 的逻辑，实现无缝跨 Bot 漫游。
