# P0 现金流主线 PRD（30 天）

## 目标
30 天内把现有代码资产变成一条可重复执行的“赚钱流水线”，覆盖：
Bot 进入 → 宠物建档 → 护理推荐 → 商品下单 → 管理员通知 → 人工成交 → 善终咨询预约。

## 范围（P0 只做什么）
### 必做（P0）
1) Bot 入口与清晰导航
2) 宠物建档（画像落库）
3) 护理推荐（AI/规则混合均可，先可用）
4) 商品下单（最小 SKU/套餐）
5) 管理员通知（新订单/新预约/新支付凭证）
6) 人工成交闭环（管理员能快速联系用户）
7) 善终咨询预约（入口 + 表单 + 预约落库）

### 不做（P0 不投入）
- Claw 复杂玩法扩展
- 高级风控矩阵（仅保底规则/冻结）
- 多城市扩张后台
- 花哨 UI 重构

## 用户路径（成交漏斗）
1) 用户从短视频/广告进入 Telegram → 打开 Claw Bot
2) 通过 3-4 个问题完成宠物建档（类型/年龄/体重/主要健康问题）
3) 用户点击“护理建议”获得推荐结论与建议套餐
4) 用户点击“购买护理包”进入 RainbowPaw（商店 MiniApp）完成下单/支付/提交凭证
5) 管理员 Telegram 收到通知（新订单 + 支付凭证），人工跟进成交
6) 用户产生善终需求时，从 Bot 进入“善终咨询预约”表单，管理员收到通知并跟进

## 功能需求（按模块）

### A. Bot 入口（Claw Bot）
- A1. /start 首屏：一句话说明 + 立即建档
- A2. 建档问答：
  - 宠物类型（猫/狗）
  - 年龄段（<1 / 1-3 / 4-7 / 7+）
  - 体重（区间）
  - 主要健康问题（单选：关节/肾脏/肠胃/皮肤/无）
- A3. 建档后主菜单：抽奖 / 护理建议 / 商城 / 善终服务

验收：新用户从进入到看到主菜单 ≤ 60 秒；画像字段能在后台/接口查询到。

### B. 建档落库（Identity）
- B1. 画像字段保存到 `identity`（global_user_id 维度）
- B2. 自动标签（例如 elder_pet）

验收：同一 telegram 用户重复进入不会产生重复用户；画像字段更新可覆盖。

### C. 护理推荐（Care Plan）
- C1. 输出格式固定：
  - rationale（一句原因）
  - plan[]（3-5 条建议）
  - recommendedPack（name/price/id）
- C2. “购买护理包”按钮跳转到 RainbowPaw 商城并打开对应分类/路径

验收：Bot 内点击“护理建议”能稳定返回推荐；点击购买能打开商店。

### D. 商品下单（Shop）
- D1. 最小 SKU/套餐：至少 2-3 个可卖护理商品/套餐
- D2. 下单流程：选择商品 → 生成订单 → 发起支付 → 提交凭证

验收：用户能在 10 分钟内完成下单并提交凭证。

### E. 管理员通知（Admin Notify）
- E1. 新订单通知：创建订单时推送管理员 Telegram（order_id、金额、联系方式/备注）
- E2. 支付凭证通知：提交凭证时推送管理员 Telegram（payment_id、proof_file 链接）
- E3. 新预约通知：创建善终预约时推送管理员 Telegram（booking_id、时间、用户）

验收：三类事件都有通知；管理员能点击链接查看信息或复制凭证。

### F. 人工成交（CRM 最小闭环）
- F1. 管理员能根据通知快速联系用户（在 Telegram 里一键发消息模板）
- F2. 记录一次“跟进结果”（成交/未成交/待跟进）

验收：每个订单至少有一次跟进记录。

### G. 善终咨询预约（Aftercare）
- G1. 入口清晰：Bot/网页都能进入预约
- G2. 预约字段：宠物信息、城市/地址、时间偏好、需求类型（火化/纪念等）

验收：预约能落库并通知管理员；后续可被 CRM 跟进。

## 关键指标（P0 只追三件事）
- 完成建档率：进入 Bot → 完成画像
- 付款意向率：建档/推荐 → 点击购买/预约
- 成交率：管理员跟进 → 付费（以凭证提交/订单状态为准）
