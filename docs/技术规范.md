# OpenClaw & RainbowPaw 重构技术规范 (V1.0)

## 一、 整体架构

本项目采用**微服务+网关/Bot路由**的混合架构：
- **Bots (Client Layer)**：基于 Node.js + `node-telegram-bot-api`，充当用户的直接交互入口。分为 `claw-bot` 和 `rainbowpaw-bot`。
- **Core Services (Service Layer)**：基于 **NestJS + TypeScript** 构建，提供底层核心业务能力。
  - `identity-service`: 统一用户身份、标签管理。
  - `wallet-service`: 统一积分、余额、流水管理。
  - `bridge-service`: Bot 间的跳转追踪与事件上报。
- **Database (Data Layer)**：单一 PostgreSQL 实例，按服务划分 Schema (`identity`, `wallet`, `bridge`)。

## 二、 代码组织规范

### 1. 目录结构模式 (Monorepo 推荐)
为了方便本地调试与共享配置，采用统一的代码仓库结构：
```text
/
├── bots/                  # Node.js 机器人项目
│   ├── src/
│   │   ├── bots/          # 各个机器人的实例
│   │   ├── routes/        # 命令路由
│   │   └── services/      # 封装对 Core Services 的 API 调用
│   └── package.json
├── services/              # NestJS 微服务集合
│   ├── identity-service/
│   ├── wallet-service/
│   └── bridge-service/
├── docs/                  # 需求与设计文档
├── docker-compose.yml     # 本地联调环境 (PostgreSQL, Redis等)
└── README.md
```

### 2. NestJS 服务规范
- **包管理器**：统一使用 `npm`。
- **全局前缀**：所有 API 必须挂载 `/api` 前缀。
- **验证与转换**：必须在 `main.ts` 开启全局 `ValidationPipe` (自带 `whitelist: true` 和 `transform: true`)。
- **响应格式**：通过统一的拦截器 (Interceptor) 或公共工具函数封装返回体，确保遵守以下契约：
  ```json
  {
    "code": 0,
    "message": "ok",
    "data": {}
  }
  ```
- **依赖注入**：严禁在 Controller 中写重业务逻辑，必须将逻辑下沉到 Service。

### 3. Bot 路由开发规范
- **无状态设计**：Bot 进程必须是无状态的。用户的当前上下文（例如处于等待输入地址的状态）应存储在 Redis 中，而不是内存变量。
- **服务解耦**：Bot **严禁直接直连 PostgreSQL 数据库**。所有对用户、积分、订单的读写，必须通过 HTTP 调用对应的 Core Service API。

## 三、 数据库与模型规范

### 1. 命名规范
- **表名与字段**：全小写蛇形命名法 (`snake_case`)。如 `global_users`, `points_locked`。
- **主键**：统一采用 `id BIGSERIAL PRIMARY KEY`。
- **业务ID**：必须使用具备业务含义的字符串前缀生成（如 `g_` 代表 global_user_id，`o_` 代表 order_id），不暴露自增主键给前端或外部服务。

### 2. 金额与精度
- 涉及金额与积分的字段，**绝对禁止使用 `FLOAT` 或 `DOUBLE`**。
- 必须使用 `NUMERIC(12,2)` 确保财务精度。
- 业务代码中计算金额时，建议转换为分为单位的整数运算，或者使用类似 `decimal.js` 的安全库。

### 3. 时间字段
- 所有表必须包含 `created_at` (TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP)。
- 需要更新的表必须包含 `updated_at`，由数据库触发器或 ORM 自动维护。
- 所有时间统一使用 UTC 存储。

## 四、 接口与网络规范

### 1. HTTP 请求规范
- 请求头：必须携带 `Content-Type: application/json`。
- 内部认证：服务间调用及 Bot 调用服务时，必须在请求头携带内部安全 Token：`Authorization: Bearer <internal-token>`。
- 幂等性：针对扣减积分 (`/wallet/spend`)、增加积分 (`/wallet/earn`) 等写操作，调用方必须传入 `X-Idempotency-Key`（如 UUID），服务端需拦截重复请求。

### 2. 异常处理
- 服务端捕获已知异常时，抛出 NestJS 的 `HttpException`，并由全局 Filter 转换为标准错误响应（如 `code: 40001`）。
- Bot 端在捕获到接口报错时，必须给用户友好的提示（如“系统繁忙，请稍后再试”），严禁把后端 Stack Trace 吐给用户。
